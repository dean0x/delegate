## 🏗️ Architecture & Design

### Key Architectural Decisions

1. **Event-Driven Task Dependencies**
   - **Why**: Consistency with existing event-driven architecture (v0.2.1)
   - **Impact**: All dependency operations emit events (TaskDependencyAdded, DependencyResolved, TaskUnblocked)
   - **Pattern**: DependencyHandler listens for TaskCompleted/Failed/Cancelled and resolves dependencies automatically

2. **DAG Validation with DFS Cycle Detection**
   - **Why**: Prevents circular dependencies that would cause deadlocks
   - **Impact**: O(V+E) cycle detection on every dependency addition
   - **Algorithm**: Depth-First Search with recursion stack tracking
   - **File**: `src/core/dependency-graph.ts:74-112`

3. **TOCTOU Protection via Synchronous Transactions**
   - **Why**: Prevent race conditions where cycle check passes but concurrent addition creates cycle
   - **Impact**: Uses better-sqlite3's synchronous `.transaction()` for atomicity
   - **Pattern**: All checks and inserts happen synchronously with no event loop yielding
   - **File**: `src/implementations/dependency-repository.ts:117-186`

4. **Result Pattern Throughout**
   - **Why**: No exceptions in business logic, explicit error handling
   - **Impact**: All operations return `Result<T, E>` type
   - **Benefit**: Compile-time error handling verification

5. **Database-First State Management**
   - **Why**: Single source of truth, automatic recovery after crashes
   - **Impact**: SQLite with WAL mode for concurrent access
   - **Tables**: `task_dependencies` with foreign keys, composite indexes

### Technology Stack

- **Language**: TypeScript 5.x (strict mode enabled)
- **Runtime**: Node.js 20.0.0+
- **MCP Protocol**: Model Context Protocol SDK
- **Data Storage**: SQLite with better-sqlite3 (WAL mode)
- **Testing**: Vitest with 82% test coverage
- **CI/CD**: GitHub Actions (automated release on merge to main)

### Key Dependencies

- `better-sqlite3`: High-performance SQLite bindings
- `@modelcontextprotocol/sdk`: MCP protocol implementation
- `zod`: Schema validation at boundaries
- `cron-parser`: Planned for v0.4.0 task scheduling

### Design Patterns in Use

1. **Repository Pattern**: SQLiteDependencyRepository, SQLiteTaskRepository
   - Encapsulates data access
   - Pure data operations, no business logic

2. **Dependency Injection**: Constructor-based DI with Container
   - All dependencies injected, no global state
   - Makes testing trivial

3. **Event-Driven Architecture**: EventBus with specialized handlers
   - DependencyHandler, QueueHandler, PersistenceHandler, WorkerHandler
   - Zero direct state management in TaskManager

4. **Result Pattern**: Explicit success/failure types
   - `Result<T, E> = { ok: true, value: T } | { ok: false, error: E }`
   - No exceptions in business logic

5. **Immutable Domain**: Readonly data structures
   - All Task and Dependency objects are readonly
   - Updates create new objects
