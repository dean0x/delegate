    - Dynamic throttling based on failures (more complex, harder to reason about)

- **Decision**: 30-minute threshold for stale task detection

## 🏗️ Architecture & Design

### Key Architectural Decisions

1. **Pure Event-Driven Architecture (v0.2.1+)**
   - **Pattern**: ALL operations (commands AND queries) go through EventBus
   - **Why**: Eliminates coupling between components, enables comprehensive profiling, improves testability
   - **Impact**: Every component communicates via events, no direct method calls
   - **Trade-offs**: Slight performance overhead for reads, more complex debugging
   - **Location**: `src/core/events/event-bus.ts`, all handlers in `src/services/handlers/`

2. **Request-Response with Correlation IDs**
   - **Pattern**: Query events use correlation IDs for tracking responses
   - **Why**: Enables async request-response over event bus, prevents handler coupling
   - **Impact**: Handlers can respond asynchronously, timeout protection (5s default)
   - **Location**: `src/core/events/event-bus.ts:200-309`

3. **Stale Task Detection (Recovery Manager)**
   - **Pattern**: 30-minute threshold for RUNNING tasks on startup
   - **Why**: Prevents fork-bomb from crashed tasks accumulating across restarts
   - **Impact**: Old tasks marked FAILED, recent tasks re-queued
   - **Incident**: 2025-10-04 fork-bomb when spawn delay was removed
   - **Location**: `src/services/recovery-manager.ts:88-175`

4. **Spawn Burst Protection (Worker Handler)**
   - **Pattern**: 50ms minimum delay between worker spawns
   - **Why**: Resource checks happen BEFORE spawn; race condition allows burst spawning
   - **Impact**: Prevents OS-level fork-bomb from simultaneous process creation
   - **Location**: `src/services/handlers/worker-handler.ts:21-63, 194-211`

5. **Handler Profiling (Event Bus)**
   - **Pattern**: Automatic timing of all event handlers with 100ms warning threshold
   - **Why**: Slow handlers block event processing; automatic detection catches issues early
   - **Impact**: Warnings logged for slow operations, metrics for all events
   - **Location**: `src/core/events/event-bus.ts:157-211`

## 💡 Context for Future Developer

### Things to Know

1. **Gotcha #1: Spawn Delay is NOT Optional**
   - DO NOT remove the 50ms spawn delay in WorkerHandler
   - It prevents fork-bombs during recovery when multiple tasks queue simultaneously
   - See comprehensive JSDoc at `src/services/handlers/worker-handler.ts:21-48`
   - Incident reference: 2025-10-04 (7 tasks spawned simultaneously → system crash)

2. **Gotcha #2: Stale Task Detection is Critical**
   - 30-minute threshold is NOT arbitrary
   - Too short: fails legitimate long-running tasks on restart
   - Too long: accumulates crashed tasks causing fork-bombs
   - See comprehensive JSDoc at `src/services/recovery-manager.ts:88-109`

3. **Gotcha #3: All Operations Must Go Through EventBus**
   - NO direct method calls between handlers
   - Use `eventBus.emit()` for fire-and-forget
   - Use `eventBus.request()` for query-response
   - This is NOT optional - architecture depends on it for profiling and decoupling

4. **Convention: Result Types, Never Throw**
   - Business logic uses `Result<T, E>` type
   - NEVER throw exceptions in business logic
   - Only infrastructure code (DB, FS, process) may throw

5. **Convention: Structured Logging**
   - All logs must be JSON-structured via Logger interface
   - Include context objects: `{taskId, workerId, duration, etc.}`
   - Use log levels appropriately: error for actual errors, warn for concerning states, debug for detailed flow

**Key Takeaways**:

1. **Never Remove Safeguards Without Understanding Why They Exist** - The minSpawnDelayMs initially seemed like arbitrary throttling, but it prevents a critical race condition. Always investigate before removing "unnecessary" code.

2. **Document the "Why" Obsessively** - Future developers (including yourself) will thank you. The comprehensive JSDoc comments in worker-handler.ts and recovery-manager.ts explain not just WHAT the code does, but WHY it must be this way.

3. **Test Edge Cases in Production-Like Conditions** - The fork-bomb only appeared when:
   - Server restarted (recovery ran)
   - Database had stale tasks (from previous test runs)
   - Spawn delay was removed (exposing race condition)
   None of these conditions existed in unit tests.

4. **Event-Driven Architecture Pays Off** - While it adds complexity, the ability to profile all handlers automatically and the loose coupling between components makes the system much more maintainable.

**Warnings for Future Self**:
