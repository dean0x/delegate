## 🏗️ Architecture & Design

### Key Architectural Decisions

1. **Pure Event-Driven Architecture (92% compliance)**
   - **Why**: Single source of truth, consistency, testability, extensibility
   - **Impact**: ALL operations go through EventBus - no direct repository access
   - **Trade-offs**: ~1-2ms overhead per query operation (acceptable)
   - **Pattern**:
     - Commands use fire-and-forget `emit()`
     - Queries use request-response `request()` with correlation IDs
   - **Files**: task-manager.ts, all handlers (persistence, queue, worker, output, query)

2. **Fork-Bomb Protection (Two-Layer)**
   - **Why**: Recovery re-queued 7 tasks simultaneously → all spawned at once → system crash
   - **Layer 1**: Stale task detection (30-min threshold in recovery-manager.ts)
   - **Layer 2**: Spawn burst prevention (50ms delay in worker-handler.ts)
   - **Impact**: Prevents resource exhaustion on startup
   - **Documentation**: Comprehensive inline docs with incident reference (2025-10-04)

3. **Result Types Instead of Throwing (65% compliance, target: 100%)**
   - **Why**: Functional error handling, no unhandled exceptions, testable
   - **Current Violations**: 21 throw statements (15 in bootstrap.ts)
   - **Impact**: Inconsistent error handling, some paths throw, others return Result
   - **Action Required**: Refactor bootstrap.ts to return Result<Container>

4. **Test Infrastructure with Factories and Test Doubles**
   - **Why**: DRY principle, consistent test data, no vi.fn() mocking
   - **Quality Score**: 95/100 (excellent infrastructure)
   - **Patterns**: TaskFactory, WorkerFactory, ConfigFactory, TestEventBus, TestLogger
   - **Files**: tests/fixtures/ (factories.ts, test-doubles.ts, test-helpers.ts, constants.ts)
