## 🏗️ Architecture & Design

### Key Architectural Decisions
1. **"Parse, Don't Validate" Configuration Pattern**
   - Why: Schema should transform input into complete, valid configuration
   - Impact: All config fields guaranteed present after parse, no optional fields in type
   - Location: `src/core/configuration.ts` with Zod `.default()` pattern
   - Reference: Alexis King's blog post (linked in code comments)

2. **Component-Level Validation (NEW)**
   - Why: Schema validation insufficient for system compatibility checks
   - Impact: Operators get actionable warnings about misconfigurations before runtime issues
   - Location: `src/core/config-validator.ts` called from `bootstrap.ts`
   - Pattern: Non-fatal warnings with severity levels and specific recommendations

3. **Worktrees as Experimental Feature (NEW)**
   - Why: Too complex for most users; opt-in reduces cognitive load
   - Impact: 95% of users get simpler UX, 5% can still enable for isolation
   - Configuration: `useWorktreesByDefault: false`, env var `USE_WORKTREES_BY_DEFAULT=true` to enable
   - Documentation: Comprehensive guide in `.docs/features/worktrees/README.md` (local only)

4. **Pure Event-Driven Architecture (Existing)**
   - Why: Loose coupling, testability, extensibility
   - Impact: All operations go through EventBus, no direct method calls between services
   - Components: EventBus, event handlers (Persistence, Queue, Worker, Output)

5. **Security-First Configuration Design (Existing)**
   - Why: Delegate runs on dedicated servers with untrusted task inputs
   - Impact: All resource limits have hard upper bounds to prevent exhaustion attacks
   - Limits: Max 32 CPU cores, 64GB memory, 1 hour timeout
