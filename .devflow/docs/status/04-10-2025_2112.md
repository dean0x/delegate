### Decisions Made

- **Decision**: Use `startedAt` instead of `updatedAt` for RUNNING task age calculation
  **Rationale**:
    - `updatedAt` is not persisted to database (schema limitation)
    - `startedAt` is the semantically correct timestamp for RUNNING tasks
    - More accurate for detecting crashed workers (time since task actually started executing)
  **Alternatives Considered**:
    - Add `updated_at` column to schema (rejected: requires migration, adds complexity)
    - Use `created_at` always (rejected: inaccurate for long-queued tasks)
  **Impact**: Stale detection now works reliably, preventing fork-bomb on startup

- **Decision**: Keep 50ms spawn delay despite successful stale detection
  **Rationale**:
    - Defense in depth: two independent protections (stale detection + spawn throttling)
    - Prevents burst spawning even if stale detection fails
    - OS-level fork protection (CPU/memory spikes from simultaneous process creation)
  **Alternatives Considered**:
    - Remove delay entirely (rejected: risk of fork-bomb if stale detection has bugs)
    - Increase to 100ms (rejected: unnecessary latency for normal operations)
  **Impact**: System remains stable even with edge cases

- **Decision**: Do not add `updated_at` column to database schema
  **Rationale**:
    - Current schema is sufficient with proper field usage
    - Migration would affect production databases
    - `startedAt`/`completedAt` provide adequate lifecycle tracking
  **Alternatives Considered**:
    - Add `updated_at` and migrate (rejected: unnecessary complexity)
    - Use separate metadata table (rejected: over-engineering)
  **Impact**: Simpler schema, faster queries, no migration needed

## 🏗️ Architecture & Design

### Key Architectural Decisions

1. **Pure Event-Driven Architecture (v0.2.1)**
   - Why: Single source of truth, consistency, testability, extensibility
   - Pattern: ALL operations (commands AND queries) go through EventBus
   - Impact: TaskManager has zero direct dependencies on Repository/OutputCapture
   - Trade-offs: ~1ms overhead for queries vs direct access (acceptable)
   - Documentation: `src/services/task-manager.ts:1-14`, `docs/architecture/EVENT_FLOW.md`

2. **Stale Task Detection (30-minute threshold)**
   - Why: Prevent fork-bomb from crashed workers on server restart
   - Implementation: `src/services/recovery-manager.ts:88-175`
   - Uses: `startedAt` timestamp for RUNNING tasks (NOT `updatedAt` - doesn't exist in DB)
   - Impact: Recent tasks (<30 min) recovered, stale tasks (>30 min) marked FAILED
   - CRITICAL: DO NOT REMOVE without implementing alternative (heartbeat system, graceful shutdown)

3. **Fork-Bomb Protection (50ms spawn delay)**
   - Why: Prevent OS-level resource exhaustion from burst process creation
   - Implementation: `src/services/handlers/worker-handler.ts:21-63, 194-211`
   - Reduced from: 100ms → 50ms (optimal balance)
   - Works with: Stale detection (defense in depth)
   - Documentation: Comprehensive JSDoc with incident reference (2025-10-04)

4. **Request-Response Event Pattern**
   - Why: Event-driven queries need synchronous-like behavior
   - Implementation: Correlation IDs, 5s timeout, Promise-based
   - Used for: TaskStatusQuery, TaskLogsQuery, NextTaskQuery
   - Impact: Maintains event-driven purity while supporting query operations
