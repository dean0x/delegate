## 🏗️ Architecture & Design

### Key Architectural Decisions
1. **Event-Driven Architecture (v0.2.1)**
   - Why: Eliminates race conditions, provides consistent coordination model, enables extensibility
   - Impact: All components communicate through EventBus; no direct state management; specialized event handlers for different concerns (PersistenceHandler, QueueHandler, WorkerHandler, OutputHandler, DependencyHandler)

2. **DAG-Based Task Dependencies (v0.3.0)**
   - Why: Enables complex workflows where tasks wait for other tasks to complete
   - Impact: Cycle detection prevents deadlocks; automatic unblocking when dependencies complete; O(V+E) DFS algorithm with TOCTOU protection via synchronous transactions

3. **Git Worktree Isolation**
   - Why: Tasks execute in isolated git worktrees to prevent conflicts and enable parallel development
   - Impact: Each task can work on separate branch without affecting main working tree; automatic PR creation and merging

4. **Autoscaling Worker Pool**
   - Why: No artificial worker limits - use all available system resources
   - Impact: Monitors CPU/memory in real-time; spawns workers when resources available; maintains 20% CPU headroom and 1GB RAM reserve

5. **SQLite Persistence with WAL Mode**
   - Why: Task queue must survive server restarts; ACID guarantees for dependency resolution
   - Impact: Automatic crash recovery on startup; concurrent read access; synchronous transactions for critical sections (cycle detection)

### Technology Stack
- **Language**: TypeScript (Node.js 20.0.0+)
- **Framework**: Model Context Protocol (MCP) SDK
- **Data Storage**: SQLite with WAL mode
- **Process Management**: Child process spawning (`claude` CLI)
- **Testing**: Vitest with coverage
- **CI/CD**: GitHub Actions with automated npm publishing

### Design Patterns in Use
- **Event-Driven Architecture**: EventBus coordinates all components via events (TaskDelegated, TaskQueued, WorkerSpawned, etc.)
- **Dependency Injection**: All services receive dependencies through constructors for testability
- **Result Types**: Business logic returns `{ ok: true, value }` or `{ ok: false, error }` instead of throwing
- **Repository Pattern**: Database access abstracted through SQLiteTaskRepository and SQLiteDependencyRepository
- **Handler Pattern**: Specialized event handlers respond to specific events (PersistenceHandler, QueueHandler, etc.)
- **Observer Pattern**: Components subscribe to EventBus and react to events
