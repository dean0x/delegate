   - **Problem**: Tests using `vi.useFakeTimers()` were hanging or timing out due to complex async interactions
   - **Solution**: Replaced fake timers with real timers using SHORT durations (100ms) from test constants
   - **Rationale**: Real timers are more reliable, tests complete faster, less brittle
   - **Impact**: EventBus request timeout tests now pass consistently

3. **QueryHandler API Breaking Change - Runtime Crash Prevention**
   - **Problem**: QueryHandler threw errors for not-found tasks, but we changed it to return null for graceful handling. TaskManager.retry() used unsafe type assertion `as Task` that would hide null values, causing `TypeError: Cannot read property 'status' of null` crashes.
   - **Solution**:
     - Updated TaskManager.retry() to explicitly check for null and return `taskNotFound()` error
     - Updated TaskManager.getStatus() with same null handling
     - Removed all unsafe type assertions
   - **Location**: `src/services/task-manager.ts:165-187, 88-107`
   - **Impact**: Prevents production crashes, type-safe null handling

4. **Event Data Structure Mismatch**
   - **Problem**: Tests expected raw payload `{ value: 'test' }`, but EventBus delivers full event with metadata `{ type, eventId, timestamp, source, value }`
   - **Solution**: Updated test expectations to use `toMatchObject()` with full event structure
   - **Impact**: Tests now validate actual API behavior

### Decisions Made

- **Decision**: Return null for not-found tasks instead of throwing errors

  - **Alternatives Considered**: Keep throwing errors (rejected: forces try/catch everywhere, less ergonomic)
  - **Migration Impact**: Required updates to all TaskStatusQuery call sites

- **Decision**: Remove fake timers from tests, use real timers with SHORT constants
  - **Rationale**: Fake timers caused deadlocks with async operations, real timers are faster and more reliable
  - **Alternatives Considered**: Fix fake timer implementation (rejected: too complex, fragile)
  - **Impact**: Tests run faster, more predictable, easier to debug

- **Decision**: Use test factories (`createTestConfiguration()`) instead of inline test data
  - **Rationale**: Consistency with TEST_STANDARDS.md, DRY principle, easier to update defaults
  - **Impact**: More maintainable tests, follows project conventions

## 🏗️ Architecture & Design

### Key Architectural Decisions

1. **Event-Driven Architecture (Pure EventBus Pattern)**
   - Why: Decouples components, makes testing easier, enables pub/sub patterns
   - Impact: ALL operations (commands AND queries) go through EventBus
   - Files: `src/core/events/event-bus.ts`, all service handlers

2. **Result Types Instead of Throwing Errors**
   - Why: Explicit error handling, type-safe, forces callers to handle errors
   - Impact: No try/catch blocks in business logic, errors are values
   - Pattern: `Result<T, E> = { ok: true; value: T } | { ok: false; error: E }`

3. **Dependency Injection Throughout**
   - Why: Makes testing trivial, enables substitution, clear dependencies
   - Impact: All services receive dependencies via constructor
   - Example: `QueryHandler(repository, outputCapture, eventBus, logger)`

4. **Parse, Don't Validate (Zod Schemas)**
   - Why: Guarantees complete objects, no undefined fields, single source of truth
   - Impact: All config fields use `.default()`, not `.optional()`
   - Files: `src/core/configuration.ts`, `src/core/config-validator.ts`
