## Issues in Your Changes (BLOCKING)

### HIGH

**Activity feed passes full channel list instead of recent channels** - `src/cli/dashboard/use-dashboard-data.ts:368`
**Confidence**: 85%
- Problem: `fetchMetricsExtras()` receives the full `channels` list (fetched via `findAll(FETCH_LIMIT)`) and passes it directly into `buildActivityFeed()`. All other entity types (tasks, loops, orchestrations, schedules, pipelines) use `findUpdatedSince(since1h, 50)` to limit to recent activity. Channels use the entire list regardless of age because `ChannelRepository` has no `findUpdatedSince` method. This asymmetry means channels always dominate the activity feed by count, and the feed shows stale channel entries that were last updated days ago alongside truly recent activity.
- Fix: Add a `findUpdatedSince(sinceMs: number, limit: number)` method to `ChannelRepository` (matching the pattern of the other 5 entity repositories), then call it in `fetchMetricsExtras` alongside the other recent-entity fetches in `Promise.all`. The comment at line 330-332 acknowledges the gap ("ChannelRepository has no findUpdatedSince") but the workaround degrades feed quality. Alternatively, filter `channels` client-side before passing to `buildActivityFeed`:
  ```typescript
  const recentChannels = channels.filter(c => (c.updatedAt ?? c.createdAt ?? 0) >= since1h);
  ```

### MEDIUM

**`ChannelMessage.id` generated with `crypto.randomUUID()` in handler rather than domain layer** - `src/services/handlers/channel-message-persistence-handler.ts:90`
**Confidence**: 82%
- Problem: The message ID is generated inside the persistence handler (`id: \`cm-${crypto.randomUUID()}\``) rather than in the domain layer or at the creation boundary. In this codebase, entity IDs are typically generated by the domain layer (e.g., `ChannelId(crypto.randomUUID())`) or by the service creating the entity. The handler is an event consumer ‚Äî it transforms event data into a persistence record. Generating IDs here makes the handler responsible for identity semantics, which is a layering concern.
- Fix: This is a minor layering issue. The handler is best-effort and display-only, so the practical impact is low. For consistency with other entity creation patterns, the ID could be derived from the event's unique coordinates (channelId + from + round + timestamp) via a deterministic hash, which would also give idempotency on duplicate event delivery. However, given the append-only nature and display-only purpose, the current approach is acceptable.

## Pre-existing Issues (Not Blocking)

### MEDIUM

**N+1 query pattern in `rowToChannel` acknowledged but unbounded** - `src/implementations/channel-repository.ts:420-424`
**Confidence**: 85%
- Problem: Every `rowToChannel()` call issues a separate `findMembersByChannelIdStmt` query. `findAll(100)` = 101 queries. The comment acknowledges this and defers optimization. The dashboard now polls this on a 1Hz cycle (via `fetchAllData`), so this is 101 queries per second when channels exist. For the dashboard use case with `FETCH_LIMIT` channels, this is a meaningful performance concern at scale.
- Impact: Pre-existing from Phase 6. Not blocking this PR.

## Suggestions (Lower Confidence)

- **`channels` optional in `BuildActivityFeedArgs` but required in `DashboardData`** - `src/cli/dashboard/activity-feed.ts:104` vs `src/cli/dashboard/types.ts:207` (Confidence: 70%) ‚Äî The `channels` field is `readonly channels?: readonly ChannelLike[]` in `BuildActivityFeedArgs` (optional, defaulted with `?? []`) but `readonly channels: readonly Channel[]` in `DashboardData` (required). This asymmetry is harmless because the activity feed is called from two sites (metrics extras and tests), but it creates a subtle inconsistency in the interface contract. Consider making both required or both optional for clarity.

- **`capturePaneContent` is a synchronous shell exec call inside a React render cycle** - `src/cli/dashboard/use-channel-pane-preview.ts:53` (Confidence: 65%) ‚Äî The `capturePaneContent` function wraps `spawnSync` (via the tmux `ExecFn`). While it is called from `setInterval` (not during render), the synchronous exec blocks the Node.js event loop for the duration of the tmux call. For the 3-second polling interval and typical <10ms exec time, this is fine in practice. If tmux ever hangs or the system is under load, the dashboard would freeze. The `fetching` guard prevents overlapping calls but not long-running ones.

- **`channelService` optionality differs from other mutation services** - `src/cli/dashboard/types.ts:59-62` (Confidence: 62%) ‚Äî `channelService` and `channelRepo` are optional in `DashboardMutationContext`, while the first four services (`orchestrationService`, `loopService`, `scheduleService`, `taskManager`) are required. This matches the `pipelineRepo` precedent (also optional), but it means mutation code paths must guard with `mutations.channelService` checks. The PR handles this consistently (e.g., `entity-mutations.ts:89`, `entity-mutations.ts:139`), so this is a style observation rather than a bug.

### Assessment

This PR adds channels as the 6th entity type to the dashboard, following the established architectural patterns with high fidelity. Key strengths:

1. **Pattern consistency** ‚Äî The channel integration follows the exact same patterns as the existing 5 entity types: PanelId union expansion, filter cycles, keyboard shortcuts, entity-mutations switch cases, detail view routing, and activity feed integration. This consistency (applies ADR-001 for channel name constraints) makes the codebase predictable.

2. **Clean layering** ‚Äî The new `ChannelMessagePersistenceHandler` follows the established event-handler factory pattern (matching `UsageCaptureHandler`). The `ChannelMessage` domain type is defined in `core/domain.ts`, the repository interface in `core/interfaces.ts`, and persistence in `implementations/`. Dependencies point inward.

3. **Proper port extension** ‚Äî `TmuxSessionManagerCorePort` gains `capturePaneContent` as a display-only method with clear documentation about session-not-found semantics. The implementation validates session names using the shared `SESSION_NAME_REGEX` guard.

4. **Best-effort degradation** ‚Äî The `capturePaneContent` prop is optional, `channelService` is optional in mutation context, and the preview hook gracefully handles `undefined` capture functions. The dashboard degrades without crashing when tmux is unavailable.

The one blocking-level issue is the activity feed asymmetry where channels use the full entity list rather than recent-only, breaking the established `findUpdatedSince` pattern used by all other entity types.

### Conditions for Approval

1. Address the activity feed channel filtering asymmetry (add client-side date filter or implement `findUpdatedSince` on `ChannelRepository`).
